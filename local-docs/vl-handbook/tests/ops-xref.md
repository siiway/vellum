# OPS xref

The worker loads `xrefmap.yml` (or `.yaml` / `.json`) from each repo's docs
root and uses it to resolve `<xref:Uid>` and `[text](xref:Uid)` references.
Unresolved uids render as a dashed monospace box so the author sees the typo.

## Autolink form

The autolink form `<xref:System.Console>` renders as
<xref:System.Console>. Hover the link — it opens in a new tab against the
URL declared in `xrefmap.yml`.

Sample uids defined in the handbook's xrefmap:

- <xref:Vellum.Handbook>
- <xref:Vellum.OPS.Tabs>
- <xref:Vellum.OPS.Includes>
- <xref:System.Console>

## Link form (custom display text)

The `[text](xref:Uid)` form lets the author override the display name. For
instance, [the .NET console class](xref:System.Console) points at the same
target as the autolink above but uses your own anchor text.

A few more:

- [Handbook](xref:Vellum.Handbook) — local repo home.
- [Tabs feature](xref:Vellum.OPS.Tabs)
- [Includes feature](xref:Vellum.OPS.Includes)

## Unresolved fallback

A uid that isn't in the xrefmap renders as a dashed monospace box so the
author sees the broken reference: <xref:Vellum.Does.Not.Exist>.

And the link form: [missing uid](xref:Vellum.Also.Missing).

## Mixed with prose

The full sentence form works inline too. When you want to reference
<xref:Vellum.OPS.Tabs> mid-sentence, the inline xref drops in cleanly.

::: tip
xref support is most useful for API docs where uids are generated by the
toolchain (DocFX, etc.) and shipped alongside the rendered HTML.
:::
